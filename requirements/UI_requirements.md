# Wisme UI/UX 実装要件定義書

## 1. デザイン・コンセプトと基本原則
「学習アプリ」という路線に寄りすぎず、**「読書体験を拡張し、自分の書いたメモを振り返る」**ための、寄り添うようなツールを目指します。

* **視覚的主役:** 常に「本の表紙」が主役。自分の書斎を眺めているような感覚をユーザーに与える。
* **トーン:** 本の表紙の色が映えるよう、ベースはオフホワイトや紙の質感を意識した落ち着いた色調。
* **ユーザー体験:** 読書のリズムを崩さない。単語検索はメモ作成の「下準備」であり、シームレスに行えること。

## 2. 画面別UI詳細要件

### A. ダッシュボード（ホーム画面）
* **最優先要素（パターンA）:** 「いま、何を読んでいますか？」という問いかけ。
    * アカウント作成後の初回ログイン時には、このアプリで何ができるかの「いい感じ」の説明UIを表示。
    * 中央に直近で書いているメモへの再開ボタンを配置し、すぐに執筆に戻れるようにする。
* **振り返りの導線:**
    * 「久しぶりに自分の書いたメモを振り返ってみませんか？」という、柔らかい表現のポップアップを表示。
    * このポップアップは、前回の更新から時間が空いた時などに「ふンわり」と出現させる。
    * クリックすると、学習画面ではなく「メモ一覧」へ遷移させる。
* **推奨図書:** 画面下部にAmazon風のカルーセル形式（表紙が横にスライド）で表示。

### B. 読書メモ作成・編集画面
* **構成:** 本の情報の入力、章ごとのメモ、本全体の感想文、単語検索のすべてを1ページ内に配置。
* **エディタ形式:** 「ブログ記事のような一つの長い文章」を章ごとに書いていくスタイル。
* **サイドバー（Gemini単語検索）:**
    * メモ入力欄の横にサイドバー（またはポップアップ）として常駐。
    * テキストボックスに分からない単語を入力して検索すると、Gemini APIから取得した意味を表示。
    * **重要:** 検索された単語は、ユーザーが保存操作をしなくても、バックグラウンドで自動的にそのメモへ紐づく。
    * 検索結果が増えて邪魔な場合は、個別に表示を消せる（非表示）機能を設ける。
* **ビジュアル:** 本全体の感想文入力欄と、章ごとの入力欄を明確に分ける。

### C. 単語帳・クイズ機能（おまけ要素）
* **立ち位置:** メモを読み返した「ついで」にやる、おまけのような位置付け。
* **クイズ形式:** フラッシュカード形式。
    * 表面：登録された単語を表示。
    * 裏面：単語の意味 ＋ 「どの本のメモで調べたか（表紙画像と引用）」を表示し、当時の文脈を思い出させる。
* **ソート:** アルファベット順、登録順の切り替え。

## 3. Claude Code へのUI実装指示

既存のバックエンド処理や基本ロジックと衝突しないよう、以下の「見た目」と「フロントエンドの挙動」に特化して実装してください。

### デザイン・スタイリング
* **レスポンシブ:** PCとスマホの両方で最適化。スマホでは単語検索パネルをスワイプアップ等で表示。
* **ローディング:** Gemini APIでの検索中、検索結果表示エリアにプログレスバーまたはスケルトンUIを表示。
* **表紙の表示:** 一覧画面では、テキストよりも本の表紙画像（3:4または2:3比率）を優先して整列させる。

### インタラクション
* **章の追加:** JavaScriptを用いて、ページ遷移なしで「章の入力欄」を追加可能にする。
* **自動紐付けの視覚化:** 単語検索が完了した際、そのメモに「紐づいた」ことがわかる控えめなインジケーターを表示。
* **フラッシュカード:** 3D反転アニメーションを実装。

### 振り返り通知
* アプリ起動時、または特定の条件（期間経過）で「メモ一覧への誘導ポップアップ」をフロントエンドで制御して表示。
